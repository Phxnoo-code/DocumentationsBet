# ตัวชี้วัดระบบ

## คำอธิบาย

หน้านี้ใช้สำหรับติดตามสถานะทรัพยากรของระบบ เช่น CPU, Memory, Database, ปริมาณ operations และจำนวน connections ตามช่วงเวลาที่เลือก เพื่อช่วยให้แอดมินตรวจสอบความผิดปกติของระบบและประเมินความเสี่ยงก่อนเกิดปัญหาได้เร็วขึ้น

## สิทธิ์การเข้าใช้งาน

การเข้าถึงหน้า **ตัวชี้วัดระบบ** ควรจำกัดเฉพาะแอดมินที่มีหน้าที่ดูแลระบบ หรือแอดมินที่ได้รับสิทธิ์ตรวจสอบข้อมูลด้านระบบและความปลอดภัย

* **สิทธิ์ระดับดูข้อมูล:** แอดมินสามารถดูคะแนนสุขภาพระบบ ตัวชี้วัดทรัพยากร กราฟแนวโน้ม และข้อมูล Insight ได้เท่านั้น
* **สิทธิ์ระดับจัดการ:** แอดมินสามารถดูข้อมูลได้เช่นเดียวกับสิทธิ์ระดับดูข้อมูล และสามารถใช้เครื่องมือ เช่น **รีเฟรช**, ปรับช่วงเวลา, กำหนดช่วงเอง และเปลี่ยนความละเอียดของข้อมูลได้ตามสิทธิ์ที่ระบบกำหนด

## วิธีเข้าถึงหน้านี้

## รายละเอียดของแต่ละส่วนในหน้าจอ

### 1. ส่วนหัวหน้าและคะแนนสุขภาพระบบ

<a href="/images/system-metrics/image1.png" target="_blank" style="display: block; margin: 20px 0;">
  <img src="/images/system-metrics/image1.png" alt="ส่วนหัวหน้าและคะแนนสุขภาพระบบ" style="width: 100%; max-width: 100%; border-radius: 8px; box-shadow: 0 2px 6px rgba(0,0,0,0.15);">
</a>

ส่วนบนของหน้าแสดงชื่อหน้า **ตัวชี้วัดระบบ** พร้อมปุ่ม **รีเฟรช** และการ์ดคะแนนสุขภาพระบบ เพื่อให้แอดมินเห็นภาพรวมความเสี่ยงของระบบได้ทันที

* **รีเฟรช:** ใช้อัปเดตข้อมูลตัวชี้วัดระบบให้เป็นข้อมูลล่าสุด
* **คะแนนสุขภาพระบบ:** แสดงคะแนนรวมของระบบจาก 100 คะแนน โดยอ้างอิงจากค่า p95 ของ CPU, Memory และ Database
* **จุดเกินเกณฑ์:** แสดงจำนวน metric ที่เกินเกณฑ์แจ้งเตือน เช่น เกิน 85%
* **Peak resource:** แสดงทรัพยากรที่มีค่าสูงสุดในช่วงเวลาที่เลือก เช่น Memory สูงสุด
* **Operations ล่าสุด:** แสดงจำนวน operations ล่าสุดของระบบ
* **วิธีอ่านข้อมูล:** อธิบายหลักการอ่านค่า utilization, saturation และแนวทางการตรวจสอบ spike

> ข้อควรทราบ: คะแนนสุขภาพระบบยิ่งต่ำ หมายถึงระบบมีความเสี่ยงหรือมีทรัพยากรบางส่วนเข้าใกล้เกณฑ์ที่ควรตรวจสอบ

### 2. ตัวกรองข้อมูลและการกำหนดช่วงเวลา

<a href="/images/system-metrics/image2.png" target="_blank" style="display: block; margin: 20px 0;">
  <img src="/images/system-metrics/image2.png" alt="ตัวกรองข้อมูลและการกำหนดช่วงเวลา" style="width: 100%; max-width: 100%; border-radius: 8px; box-shadow: 0 2px 6px rgba(0,0,0,0.15);">
</a>

ส่วน **ตัวกรอง** ใช้กำหนดช่วงเวลาที่ต้องการตรวจสอบ ความละเอียดของข้อมูล และเกณฑ์แจ้งเตือน เพื่อให้กราฟและการ์ดสรุปแสดงข้อมูลตามเงื่อนไขที่เลือก

* **ช่วงเวลาเร็ว:** ใช้เลือกช่วงเวลาสำเร็จรูป เช่น **1 ชม.**, **6 ชม.**, **24 ชม.** และ **7 วัน**
* **กำหนดเอง:** ใช้เปิดตัวเลือกสำหรับกำหนดวันและเวลาเริ่มต้น-สิ้นสุดเอง
* **ความละเอียด:** ใช้กำหนดความถี่ของข้อมูลที่แสดงในกราฟ เช่น ทุก 5 นาที
* **เกณฑ์แจ้งเตือน:** ใช้กำหนดระดับเปอร์เซ็นต์ที่ระบบใช้เป็นเส้นอ้างอิง เช่น 85%
* **วันที่เริ่มต้น / เวลา:** ใช้กำหนดจุดเริ่มต้นของข้อมูล
* **วันที่สิ้นสุด / เวลา:** ใช้กำหนดจุดสิ้นสุดของข้อมูล

> ข้อควรระวัง: หากเลือกช่วงเวลายาวมากแต่ตั้งความละเอียดถี่เกินไป อาจทำให้กราฟมีข้อมูลจำนวนมากและอ่านยากขึ้น

### 3. การ์ดสรุป CPU, Memory และ Database

<a href="/images/system-metrics/image3.png" target="_blank" style="display: block; margin: 20px 0;">
  <img src="/images/system-metrics/image3.png" alt="การ์ดสรุป CPU Memory และ Database" style="width: 100%; max-width: 100%; border-radius: 8px; box-shadow: 0 2px 6px rgba(0,0,0,0.15);">
</a>

ส่วนนี้แสดงค่าการใช้งานทรัพยากรหลักของระบบแยกเป็น 3 การ์ด ได้แก่ **CPU**, **Memory** และ **Database** เพื่อให้แอดมินตรวจสอบสถานะล่าสุดและค่าทางสถิติได้รวดเร็ว

* **CPU:** แสดงการใช้งาน CPU ล่าสุด ค่าเฉลี่ย ค่า p95 และค่าสูงสุด
* **Memory:** แสดงการใช้งาน Memory ล่าสุด ค่าเฉลี่ย ค่า p95 และค่าสูงสุด
* **Database:** แสดงการใช้งาน Database ล่าสุด ค่าเฉลี่ย ค่า p95 และค่าสูงสุด
* **ล่าสุด:** แสดงค่าการใช้งานล่าสุดของ metric นั้น
* **เฉลี่ย:** แสดงค่าเฉลี่ยตลอดช่วงเวลาที่เลือก
* **p95:** แสดงค่าที่ใช้ประเมินการใช้งานช่วงสูง โดยตัดค่าผิดปกติบางส่วนออก
* **สูงสุด:** แสดงค่าสูงสุดที่พบในช่วงเวลาที่เลือก
* **เกณฑ์เตือน:** แสดงเกณฑ์ที่ใช้ตรวจจับความเสี่ยง
* **ไม่พบ spike:** แสดงว่าไม่พบช่วงเวลาที่ค่าพุ่งผิดปกติเกินเกณฑ์

> ข้อควรทราบ: ค่า p95 เหมาะสำหรับดูแนวโน้มการใช้งานจริงมากกว่าการดูค่าสูงสุดเพียงค่าเดียว เพราะช่วยลดผลกระทบจากค่าพุ่งสั้น ๆ

### 4. แนวโน้มการใช้ทรัพยากร

<a href="/images/system-metrics/image4.png" target="_blank" style="display: block; margin: 20px 0;">
  <img src="/images/system-metrics/image4.png" alt="แนวโน้มการใช้ทรัพยากร" style="width: 100%; max-width: 100%; border-radius: 8px; box-shadow: 0 2px 6px rgba(0,0,0,0.15);">
</a>

กราฟ **แนวโน้มการใช้ทรัพยากร** ใช้ดูการเปลี่ยนแปลงของ CPU, Memory และ Database ตามช่วงเวลาที่เลือก เพื่อให้เห็นว่าทรัพยากรใดมีแนวโน้มสูงขึ้นหรือต่ำลง

* **มุมมอง:** ใช้เลือกประเภทข้อมูลที่ต้องการดู เช่น ภาพรวม
* **CPU:** แสดงด้วยเส้นหรือสัญลักษณ์สีเหลือง
* **Memory:** แสดงด้วยเส้นหรือพื้นที่สีฟ้า
* **Database:** แสดงด้วยเส้นหรือสัญลักษณ์สีเขียว
* **เกณฑ์แจ้งเตือน:** แสดงเป็นเส้นอ้างอิง เช่น 85% เพื่อดูว่าทรัพยากรเข้าใกล้ระดับที่ต้องระวังหรือไม่
* **แกนเวลา:** แสดงช่วงเวลาตามตัวกรองที่เลือก
* **แกนเปอร์เซ็นต์:** แสดงระดับการใช้งานของทรัพยากรเป็นเปอร์เซ็นต์

> ข้อควรระวัง: หากเส้นของทรัพยากรใดอยู่ใกล้หรือเกิน **เกณฑ์แจ้งเตือน** ต่อเนื่อง ควรตรวจสอบโหลดระบบและพฤติกรรมการใช้งานในช่วงเวลาดังกล่าวเพิ่มเติม

### 5. Insight ตามช่วงเวลานี้

<a href="/images/system-metrics/image5.png" target="_blank" style="display: block; margin: 20px 0;">
  <img src="/images/system-metrics/image5.png" alt="Insight ตามช่วงเวลานี้" style="width: 100%; max-width: 100%; border-radius: 8px; box-shadow: 0 2px 6px rgba(0,0,0,0.15);">
</a>

ส่วน **Insight ตามช่วงเวลานี้** ใช้สรุปค่าที่สำคัญจากข้อมูลในช่วงเวลาที่เลือก เพื่อให้แอดมินเห็นประเด็นที่ควรสนใจโดยไม่ต้องอ่านกราฟทั้งหมด

* **CPU:** แสดงค่า p95 พร้อมข้อมูลเปรียบเทียบกับจุดก่อนหน้า และค่าต่ำสุด
* **Memory:** แสดงค่า p95 พร้อมแนวโน้มการเพิ่มหรือลดลงจากจุดก่อนหน้า
* **Database:** แสดงค่า p95 ของ Database
* **Connections เฉลี่ย:** แสดงจำนวน connections เฉลี่ยในช่วงเวลาที่เลือก
* **Operations ล่าสุด:** แสดงจำนวน operations ล่าสุด พร้อมค่าเฉลี่ยต่อช่วงข้อมูล

> ข้อควรทราบ: หากค่า Memory หรือทรัพยากรอื่นมีค่า p95 สูง ควรดูร่วมกับกราฟแนวโน้มและข้อมูลโหลดระบบ เพื่อประเมินว่าค่าสูงเกิดขึ้นชั่วคราวหรือเกิดขึ้นต่อเนื่อง

### 6. โหลดระบบและจำนวน connection

<a href="/images/system-metrics/image6.png" target="_blank" style="display: block; margin: 20px 0;">
  <img src="/images/system-metrics/image6.png" alt="โหลดระบบและจำนวน connection" style="width: 100%; max-width: 100%; border-radius: 8px; box-shadow: 0 2px 6px rgba(0,0,0,0.15);">
</a>

กราฟ **โหลดระบบและจำนวน connection** ใช้ดูปริมาณงานที่ระบบประมวลผลและจำนวน connections ในช่วงเวลาที่เลือก เพื่อประเมินความหนาแน่นของการใช้งานระบบ

* **Operations:** แสดงปริมาณงานหรือจำนวน operations ของระบบ มักแสดงเป็นแท่งสีฟ้า
* **Connections:** แสดงจำนวน connections ที่เชื่อมต่อกับระบบ มักแสดงเป็นเส้นสีเขียว
* **ล่าสุด:** แสดงค่า operations ล่าสุดของช่วงข้อมูล
* **แกนเวลา:** แสดงช่วงเวลาที่ใช้วิเคราะห์
* **แกนจำนวน:** แสดงปริมาณ operations และจำนวน connections ตามสเกลของกราฟ

> ข้อควรระวัง: หาก Operations หรือ Connections เพิ่มขึ้นผิดปกติพร้อมกับ CPU, Memory หรือ Database สูงขึ้น ควรตรวจสอบโหลดระบบในช่วงเวลานั้นทันที

## ข้อควรระวังสำคัญ

* ควรตรวจสอบ **คะแนนสุขภาพระบบ** ร่วมกับการ์ด CPU, Memory และ Database ไม่ควรดูค่าใดค่าหนึ่งเพียงอย่างเดียว
* หากค่า p95 สูงใกล้ **เกณฑ์แจ้งเตือน** ควรตรวจสอบกราฟย้อนหลังเพื่อดูว่าเป็นค่าสูงชั่วคราวหรือสูงต่อเนื่อง
* หาก Memory สูงต่อเนื่อง อาจส่งผลต่อเสถียรภาพของระบบ ควรตรวจสอบโหลดและการใช้งานในช่วงเวลานั้น
* การเลือกช่วงเวลายาวเกินไปอาจทำให้รายละเอียดบางช่วงถูกเฉลี่ยจนมองเห็น spike ได้ไม่ชัด
* ควรใช้ข้อมูลหน้านี้ประกอบกับ log หรือระบบ monitoring อื่น หากต้องวิเคราะห์ปัญหาเชิงลึก

## ปัญหาที่พบบ่อย (FAQ)

**1. ทำไมคะแนนสุขภาพระบบต่ำ ?**

* **สาเหตุ:** อาจมีทรัพยากรบางส่วน เช่น CPU, Memory หรือ Database มีค่า p95 สูง หรือเข้าใกล้เกณฑ์แจ้งเตือน
* **วิธีแก้ไข:** ตรวจสอบการ์ดสรุปแต่ละ metric และดูกราฟแนวโน้มเพื่อหาช่วงเวลาที่ค่าพุ่งสูง

**2. ทำไม Memory สูง แต่ CPU และ Database ต่ำ ?**

* **สาเหตุ:** ระบบอาจใช้หน่วยความจำมากจาก process, cache หรือโหลดบางประเภทที่ไม่กระทบ CPU โดยตรง
* **วิธีแก้ไข:** ตรวจสอบช่วงเวลาที่ Memory สูง และดูว่าเกิดพร้อมกับ operations หรือ connections เพิ่มขึ้นหรือไม่

**3. ทำไมกราฟไม่แสดง spike ทั้งที่ค่าสูงสุดดูมาก ?**

* **สาเหตุ:** ค่าสูงสุดอาจเกิดขึ้นสั้นมาก หรือช่วงความละเอียดของข้อมูลอาจทำให้กราฟแสดงเป็นค่าเฉลี่ย
* **วิธีแก้ไข:** ลดช่วงเวลาให้แคบลง หรือปรับความละเอียดให้ละเอียดขึ้นเพื่อดูข้อมูลเฉพาะช่วง

**4. ทำไมค่า Operations สูงแต่คะแนนสุขภาพยังไม่แย่มาก ?**

* **สาเหตุ:** Operations สูงไม่ได้หมายความว่าทรัพยากรเกินเกณฑ์เสมอไป หาก CPU, Memory และ Database ยังรับโหลดได้ คะแนนอาจยังไม่ลดมาก
* **วิธีแก้ไข:** ดู Operations ร่วมกับ CPU, Memory, Database และ Connections เพื่อประเมินผลกระทบจริง

**5. ควรเลือกช่วงเวลาแบบใดในการตรวจสอบปัญหา ?**

* **สาเหตุ:** ช่วงเวลาที่กว้างหรือแคบเกินไปอาจทำให้วิเคราะห์ปัญหาได้ไม่ชัด
* **วิธีแก้ไข:** หากต้องการดูปัญหาล่าสุดให้เลือก **1 ชม.** หรือ **6 ชม.** หากต้องการดูแนวโน้มระยะยาวให้เลือก **24 ชม.** หรือ **7 วัน**
